## 数据库设计

### 表结构

#### 1. `power_records` - 电费记录表

存储从西华大学电费系统抓取的电费数据，包括余量和用电量信息。支持多宿舍监控，每个宿舍通过dorm_number和room_id关联。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键ID，自增 |
| dorm_number | VARCHAR(50) | 宿舍号（如：320、324），用于标识不同的宿舍 |
| balance | FLOAT | 电费余量（度），主要监控项，通常是空调余量，用于判断是否需要告警 |
| kbalance | FLOAT | 空调余量（度），从API获取的空调专用电费余量 |
| zbalance | FLOAT | 照明余量（度），从API获取的照明专用电费余量 |
| kpower_consumption | FLOAT | 空调用电量（度），与上次记录的差值，表示本次记录周期内的空调用电量 |
| zpower_consumption | FLOAT | 照明用电量（度），与上次记录的差值，表示本次记录周期内的照明用电量 |
| power_consumption | FLOAT | 用电量（度），已废弃，保留用于兼容性，请使用kpower_consumption和zpower_consumption |
| record_time | DATETIME | 记录时间，数据抓取的时间点 |
| created_at | DATETIME | 创建时间，记录插入数据库的时间 |

#### 2. `alert_rules` - 告警规则表

存储每个宿舍的告警配置，包括阈值设置和告警方式（邮件/QQ）。每个宿舍对应一条规则，通过dorm_number唯一标识。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键ID，自增 |
| dorm_number | VARCHAR(50) | 宿舍号（如：320、324），唯一标识一个宿舍的告警规则 |
| room_id | VARCHAR(50) | 房间ID（roomid），从西华大学电费系统API获取，用于查询该宿舍的电费数据，多宿舍监控必需 |
| kthreshold | FLOAT | 空调告警阈值（度），当空调余量低于此值时触发告警 |
| zthreshold | FLOAT | 照明告警阈值（度），当照明余量低于此值时触发告警 |
| threshold | FLOAT | 告警阈值（度），已废弃，保留用于兼容性，请使用kthreshold和zthreshold |
| enabled | BOOLEAN | 是否启用告警规则，False时不会触发任何告警 |
| email_enabled | BOOLEAN | 是否启用邮件告警，True时当余量低于阈值会发送邮件 |
| email_address | VARCHAR(255) | 邮件告警接收邮箱地址，启用邮件告警时必须填写 |
| qq_enabled | BOOLEAN | 是否启用QQ告警，True时当余量低于阈值会发送QQ消息 |
| qq_receiver_id | VARCHAR(50) | 已废弃，告警群号请用 QQ_BOT_GROUP_ID |
| last_alert_time | DATETIME | 最后告警时间，用于防止频繁告警，记录最近一次成功发送告警的时间 |
| created_at | DATETIME | 创建时间，规则创建的时间 |
| updated_at | DATETIME | 更新时间，规则最后修改的时间 |

#### 3. `alert_logs` - 告警日志表

记录所有告警发送的历史记录，包括成功和失败的告警，用于审计和问题排查。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键ID，自增 |
| dorm_number | VARCHAR(50) | 宿舍号（如：320、324），标识触发告警的宿舍 |
| alert_category | VARCHAR(20) | 告警类别：ac（空调）/light（照明），标识是哪个类型的电费余量触发了告警 |
| balance | FLOAT | 触发告警时的余量（度），记录告警触发时的实际电费余量 |
| threshold | FLOAT | 告警阈值（度），记录触发告警时使用的阈值 |
| alert_type | VARCHAR(20) | 告警类型：email（邮件告警）/qq（QQ告警），标识使用的告警方式 |
| alert_status | VARCHAR(20) | 告警状态：success（发送成功）/failed（发送失败），标识告警是否成功发送 |
| alert_message | TEXT | 告警消息内容，记录实际发送的告警消息文本 |
| created_at | DATETIME | 创建时间，告警发送的时间 |

```python
def crawl_and_save(db: Session, force_alert: bool = False) -> bool:
    # 获取唯一启用的告警规则（最多一个）

        return False
    data = crawler.fetch_power_data(
        dorm_number=enabled_rule.dorm_number, 
        room_id=enabled_rule.room_id

```

## 告警系统

### 告警类型

1. **邮件告警**
   - 发送方：全局配置 `EMAIL_FROM`
   - 接收方：告警规则中的 `email_address`
   - 必须配置：启用邮件告警时必须填写接收邮箱

2. **QQ告警**
   - 机器人QQ：固定 `1270667498`（NapCat 登录号，不可改）
   - 告警群：系统配置 `QQ_BOT_GROUP_ID`（默认 6011223303，可改）
   - 仅通过群消息发送，不支持私聊

### 防频繁告警机制

- **时间窗口**: 1小时内不重复告警
- **独立控制**: 每个告警类型（email/qq）和类别（ac/light）独立控制
- **失败重试**: 如果上次告警失败，允许立即重试
- **强制告警**: 手动触发时忽略防频繁限制

## QQ机器人集成

### 技术方案

- **框架**: NoneBot（Python OneBot框架）
- **实现**: NapCatQQ（OneBot V11实现）
- **协议**: OneBot V11（WebSocket反向连接）

### 连接方式

```
NapCatQQ ──WebSocket──▶ NoneBot ──HTTP API──▶ 后端服务
```

### 配置说明

**NoneBot配置** (`backend/nonebot_bot/bot.py`):
- 监听端口：8080
- WebSocket路径：`/onebot/v11/ws`

**NapCatQQ配置** (`onebot11.json`):
```json
{
  "connections": [{
    "type": "reverse-ws",
    "url": "ws://localhost:8080/onebot/v11/ws"
  }]
}
```

NoneBot提供的HTTP API：

## 项目启动顺序

### 后端是否会自动启动 NoneBot？

**不会。** 后端服务（`backend/run.py`）只启动 FastAPI 应用（uvicorn，端口 8000），不包含 NoneBot。NoneBot 是**独立进程**，需在 `backend/nonebot_bot` 目录下单独执行 `python bot.py` 启动（端口 8080）。

### 推荐启动顺序

按依赖关系，建议按以下顺序启动：

| 顺序 | 服务 | 端口 | 说明 |
|------|------|------|------|
| 1 | **后端** | 8000 | 提供 API、定时任务、爬虫；前端和告警都依赖它 |
| 2 | **NoneBot** | 8080 | QQ 机器人 HTTP API；后端发 QQ 告警时请求此服务 |
| 3 | **NapCatQQ** | - | 主动连接 NoneBot（WebSocket），需在 NoneBot 启动后启动并登录 QQ |
| 4 | **前端** | 3000 | 请求后端 API，可最后启动 |

### 依赖关系简述

- **前端** → 请求后端 API（`/api/*`）
- **后端（告警）** → 请求 NoneBot API（`http://localhost:8080/api/send_*`）
- **NoneBot** ← 被 NapCatQQ 通过 WebSocket 连接

### 一键启动

项目根目录的 `start-all-complete.bat` 会按上述顺序自动启动：

1. 停止已有服务（后端、前端、NoneBot、NapCatQQ）
2. 检查/安装依赖
3. 启动后端（Step 4）
4. 启动 NoneBot（Step 5）
5. 启动 NapCatQQ + 前端（Step 6）

NapCatQQ 启动后需在窗口中扫码或输入账号密码登录，登录成功后会连上 NoneBot。

### 仅启动部分服务时

- **只跑前端+后端**：先启动后端，再启动前端即可；QQ 告警不可用。
- **需要 QQ 告警**：后端、NoneBot、NapCatQQ 都要运行，且 NapCatQQ 已登录并连接 NoneBot。

### 云服务器部署（含 NoneBot + NapCatQQ）

在云上 24 小时运行时可同时部署 NoneBot 和 NapCatQQ：

- **NoneBot**：与后端同机，用 systemd 管理，监听 `0.0.0.0:8080`。
- **NapCatQQ**：推荐用 **Docker** 部署，提供 WebUI（端口 6099）用于扫码登录；在 WebUI 或配置中填写反向 WebSocket 地址 `ws://127.0.0.1:8080/onebot/v11/ws` 连接本机 NoneBot。

详细步骤（依赖安装、systemd 示例、Docker 命令、防火墙与扫码问题处理）见 **README.md → 部署指南 → 云服务器部署 → （可选）云上同时部署 NoneBot + NapCatQQ**。

## 单一宿舍监控

### 实现原理

本项目仅支持单一宿舍监控，宿舍号从配置文件（`CRAWLER_DORM_NUMBER`）读取，告警规则最多存在一个。

### 配置说明

在 `backend/.env` 文件中配置：

```env
CRAWLER_DORM_NUMBER=320  # 监控的宿舍号
```

### 告警规则

- 告警规则最多存在一个
- 宿舍号必须与配置文件中的 `CRAWLER_DORM_NUMBER` 一致
- 告警规则可在监控面板中直接编辑

### 原实现原理（已废弃）

1. **配置文件设置**
   - 在 `backend/.env` 中设置 `CRAWLER_DORM_NUMBER`（宿舍号）
   - 宿舍号从配置文件读取，确保唯一性

2. **告警规则配置**
   - 告警规则最多存在一个
   - 规则中包含 `room_id`（房间ID）和 `dorm_number`（宿舍号）
   - 宿舍号必须与配置文件一致

3. **爬虫获取数据**
   - 爬虫服务获取唯一启用的告警规则
   - 使用规则中的 `room_id` 获取电费数据

4. **数据存储**
   - 单一宿舍的数据存储
   - 通过 `dorm_number` 字段标识

### 使用流程

1. **配置宿舍号**
   - 在 `backend/.env` 中设置 `CRAWLER_DORM_NUMBER=320`

2. **创建告警规则**
   - 在监控面板中创建告警规则
   - 填写房间ID（通过抓包获取）
   - 配置告警阈值和接收方

3. **自动获取数据**
   - 定时任务触发
   - 使用告警规则中的 `room_id` 获取电费数据

4. **监控面板显示**
   - 显示当前配置的宿舍信息
   - 可在监控面板直接编辑告警规则

## 定时任务

### 调度配置

- **调度器**: APScheduler
- **执行时间**: 每天 8:00, 12:00, 18:00, 22:00（可配置）
- **任务**: 爬取电费数据并检查告警

### 任务流程

1. 获取所有启用的告警规则
2. 遍历每个规则，获取电费数据
3. 保存数据到数据库
4. 检查是否需要告警
5. 发送告警（如果满足条件）
